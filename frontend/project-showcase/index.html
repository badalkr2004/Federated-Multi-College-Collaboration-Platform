    Q5: Federated Multi-College Collaboration Platform

    Federated collaboration for colleges, teams, and project builders.

    A multi-tenant platform connecting colleges while preserving data isolation, scalable API design, and intelligent user-project matchmaking.

    Backend: Complete API: Express + Drizzle Presenter: Badal Kumar

    Problem Statement

    Connect colleges without mixing their private data.

    Q5 Requirement

    Design a federated, multi-tenant collaboration platform connecting multiple colleges while ensuring data isolation, scalability, and intelligent matchmaking between users and projects.

    IsolationHost-based tenant resolution with college API key validation and JWT tenant checks.
    CollaborationProjects, members, messages, ratings, and cross-college team formation.
    MatchmakingJaccard skill overlap plus normalized reputation scoring.
    ScaleNeon PostgreSQL, Redis caching, rate limiting, stateless JWT, modular routes.

    Product Solution

    OneCollab: a tenant-aware network for student project teams.

    01College onboarding

    Colleges request access; platform admin approves and provisions API keys.

    02Tenant auth

    Students register/login through their college domain and key.

    03Project discovery

    Students browse own-college and cross-college projects.

    04Smart matching

    Skills and reputation rank projects and teammates.

    05Team workflow

    Members join, message, collaborate, and rate peers.

    Intelligent Matchmaking

    We engineered a dual-algorithm approach to connect the right talent with the most suitable projects.

    Match Score = 0.7 × Jaccard Index + 0.3 × Reputation Score
    1. Jaccard Similarity Algorithm We compute technical compatibility using the Jaccard Index. While we evaluated vector similarity search for semantic matching, we strategically deployed Jaccard similarity for this MVP. This pragmatic engineering decision delivers an elegant, highly performant, and perfectly transparent skill-matching engine optimized for a short-duration hackathon.
    2. Reputation Scoring Algorithm To establish trust across the platform, we implemented a composite reputation model. It dynamically normalizes historical peer ratings, rewarding consistent, highly-rated collaboration over time.
    Performance Optimization By utilizing an in-memory Redis cache for these computationally intensive sorting algorithms, we deliver sub-millisecond, instantaneous ranking results to the end-user.

    Security & Isolation

    Defense layers for tenant-safe collaboration.

    01Tenant resolver

    Host maps to an active college. Unknown domains are rejected.

    02College API key

    Register/login routes validate X-College-Key against the resolved tenant.

    03JWT claims

    Authenticated routes compare user token tenant with request tenant context.

    04Access rules

    Private projects remain within college; cross-college projects opt into federation.

    05Rate limits

    Onboarding, auth, and API requests have separate throttles.

    06Validation

    Zod schemas validate every request body before service logic.
